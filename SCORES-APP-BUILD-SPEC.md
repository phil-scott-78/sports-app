# Scores App — Build Spec (Design → ESPN Data)

> **What this is.** A build spec that takes the [`8c0e9190` screen design](https://claude.ai/design/p/8c0e9190-964c-42cc-b923-fdda7eb6046b) — Home Feed, Live Game Detail (8 sports), Event List (8 play-by-play archetypes), Box Score & stat tables, Team & Player overviews, Tournaments (4 bracket grammars), Standings & Following, Venue & Circuit facts — and answers, for every pixel that carries data: **which ESPN endpoint lights it up, which exact field feeds it, at what fetch cost, per which sports, and what to do when a sport doesn't have it.** It then gives the **discovery methodology** so a sport/league the design never drew still renders.
>
> **How it's grounded.** Every field path below was checked against `schema/espn-guide/` — the *observed* field guide generated by crawling real ESPN responses (2,864 responses, 34 leagues, 17 sports). Nothing here is guessed from documentation. The per-screen mappings were produced by one agent per screen and then **adversarially verified** by a second agent against the guide; the corrections are folded in and flagged inline as `[verified]` / `NOT OBSERVED`. It deliberately **ignores the existing app implementation** and reasons from the design's needs and what ESPN actually serves — though the recommended normalized model happens to equal the repo's locked `schema/canonical.ts`, which is corroboration that the design is achievable.
>
> **How to read it.** Part I is the architecture every screen shares. Part II is the nine screens, each a data-contract table. Part III is the cross-cutting/platform layer (assets, calendar, odds, notifications, degraded states, appendices). Part IV is the discovery methodology. Part V is the flat endpoint reference. The Gaps ledger at the end is the honest list of what the design asks for that ESPN can't cleanly give.

**Notation.** `events[].competitions[].competitors[].team.color` is a dotted path into a JSON response; `[]` = array element, `{key}` = an id-keyed dict. **Scores are STRINGS** (`"103"`). Tiers: **`scoreboard`** (cheap, polled), **`summary`** (rich, on detail open), **`core`** (`sports.core.api.espn.com/v2` hypermedia graph), plus `team` / `standings` / `rankings` / `golf` (web playersummary) / `device` (on-device, not ESPN).

# PART I — ARCHITECTURE (the spine every screen shares)

## 1. The one idea: discriminator-driven rendering

The design shows ~17 sports but is really **one card grammar reused**: a score/event block → chip nav → a *situation card* that changes shape → a running event feed → dense stat tables → a bracket shell. That is only buildable if **no consuming code ever branches on sport name.** Instead, every competition carries three **discriminators** that tell each widget how to read the otherwise-optional fields:

| Discriminator | Values | Decides |
|---|---|---|
| **`layout`** | `headToHead` (2 sides) · `field` (N ranked by `order`) | score block vs leaderboard/lanes |
| **`scoreKind`** | `numeric` · `toPar` (golf) · `cricket` (`161/5 (18 ov)`) · `none` (racing/MMA) | how `competitor.score` prints |
| **`competitorKind`** | `team` · `athlete` · `pair` (doubles) | identity cell: logo+color vs headshot+flag |

Plus a normalized **`phase`** (`scheduled\|live\|final\|postponed\|suspended\|canceled\|abandoned\|delayed\|unknown`) derived from `status.type.name` (never the raw `state` — a postponed game can read `state:'post'`), and a **`periodUnit`** (`quarter\|inning\|set\|round\|lap\|over_innings\|hole_rounds\|…`) that labels the raw period integer.

Then the *situation card* — the sport's flourish — **dispatches on data presence, never sport name**: `situation.balls`/`onFirst` → bases diamond; `down`/`distance` → drive field; `competition.details[]` (goal/card feed) → soccer timeline; a cricket target → chase equation; `layout==='field'` → leaderboard/lanes; **nothing applicable → no card.** A sport ESPN newly starts populating lights up for free.

> **This is the load-bearing invariant.** Classify a new league into `layout·scoreKind·competitorKind` and it inherits every existing renderer with zero new UI code (Part IV).

## 2. The three fetch tiers

| Tier | URL template | Cost | Feeds |
|---|---|---|---|
| **scoreboard** | `site.api.espn.com/apis/site/v2/sports/{sport}/{league}/scoreboard?dates=YYYYMMDD[-YYYYMMDD]` | **cheap — polled every cycle** | Home feed, league sections, glance situation, series pips, probables, odds, form, per-period line scores |
| **summary** | `.../{sport}/{league}/summary?event={id}` | **rich — one fetch when a detail opens** | Box scores, play-by-play/commentary, lineups, drives, win probability, penalty shootouts, cricket scorecard |
| **core graph** | `sports.core.api.espn.com/v2/sports/{sport}/leagues/{league}/…` (`$ref` hypermedia) | **targeted, lazy** | Standings, odds, win-prob/predictor, drives, plays, athlete season stats, tournament brackets, tennis set scores, golf course/tournament, injuries |

**Traps that pin the tiers** (all verified in the guide):

- **Standings is the CORE path** `apis/v2/.../standings`, *not* `apis/site/v2/...` (the site path returns a `{fullViewLink}` stub). Omit `?season` — ESPN's default is the true current season.
- **`summary` genuinely 404s for `golf`, `mma`, `tennis`, `racing`** → those detail screens build from scoreboard + core only (golf adds the web `playersummary`). Check `espn-guide/by-sport/index.md` before designing anything on summary.
- **Per-quarter/period line scores are already on the cheap scoreboard** (`competitors[].linescores[].value`, 99%). Render the grid from there so it survives a summary failure; upgrade to `summary.header…linescores[]` when it arrives.
- **Tennis nests one level deeper**: `events[].groupings[].competitions[].competitors[]`, and its competition id ≠ event id (resolve set scores via the competition `$ref` / `core-competitor-linescores`). A flat-path reader sees zero tennis competitors.
- **Never assume `competitions.length === 1`** — a Grand Prix weekend is several competitions (FP1/Qual/Race) under one event.

## 3. Freshness & the fetch-budget rule

**Polling** is per-device, foreground-only; ESPN's CDN is the shared cache. Cadence keys off `anyLive`: **~15s live / 30s near kickoff / 60s idle.** "Near kickoff" is derived on-device (`min` upcoming `events[].date` with `status.type.state=='pre'`) — **there is no served `nextStart` field.**

**The fetch-budget rule (non-negotiable):** *the cheap scoreboard poll is the only hot path.* Every `summary`/`core` fetch is **lazy, opportunistic, concurrency-capped, and cached with an explicit TTL** — never on the poll loop. This matters because many screens *want* a datum that is core-only; those must be deferred to on-detail-open, not fetched per poll per card. The N+1 core fan-outs the design implies, and their required trigger:

| N+1 fan-out | Cheap-tier truth | Correct trigger |
|---|---|---|
| Win probability per hero card | scoreboard has it only for **basketball** (`situation.lastPlay.probability`) | on detail open (core `predictor`/`probabilities`); **omit on the feed** for other sports |
| Football down & distance / drive | **NOT on scoreboard at all** (`situation` is baseball+basketball only) | core `situation` on detail open; never on the feed poll |
| NBA bonus / timeouts / possession arrow | core `situation` / summary header only | on detail open |
| March Madness seeds (1–16) | `curatedRank` is the AP poll rank, **not the bracket seed** | per-event core `tournamentMatchup.seed` on bracket open |
| Team season leaders / per-stat league rank | core team-statistics (+ athlete `$ref` resolve) | on team page open |
| Per-event odds line / O-U | inline on ~25% of scoreboard events near game time | on team/detail open (core odds) when not in the slate |
| Tennis set scores (competition id ≠ event id) | scoreboard groupings usually carry them | core `competitor-linescores` only as fallback |

Verify the "megaweek" (every league live at once) scenario against the concurrency cap.

## 4. The normalized model (the on-device shield)

Normalize every sport into ONE envelope on-device so the UI never sees ESPN's raw shape and the provider can be swapped. This is the recommended target shape (it equals `schema/canonical.ts`):

- **`ScoresResponse`** `{ sport, league, leagueId, leagueName, season, day, anyLive, calendarDays[], events[] }` — the feed payload.
- **`Competition`** `{ layout, scoreKind, competitorKind, status{phase,live,ended,period,periodLabel,clock}, periods{unit,regulation,played,isOvertime}, competitors[], events[] (scoring timeline), situation (live glance), meta{series,round,cut,flag,cricketClass,golf}, decision, venue, weather, attendance, headline }`.
- **`Competitor`** `{ kind, id, displayName, abbreviation, logo, logoDark, color, altColor, athletes[], homeAway, order, rank, seed, winner, score{display,value,toPar,strokes,cricket}, periodScores[], records[], shootoutScore, aggregateScore, advance, leaders[], probables[], form, hits, errors, serving }`.
- **`GameSummary`** (rich) `{ teamStats[], boxGroups[] (per-player tables), scoringPlays[]/plays[], periodLines, lineups[], winProbability, drives[], cricketInnings[], bouts[], atBats[], timeline[], recentForm[], injuries[], officials[] }`.
- Plus **`StandingsResponse`**, **`TeamCardResponse`/`TeamDetailResponse`**, **`RankingsResponse`**, **`GolfScorecardResponse`** — each a thin, generic shape so one widget set renders every sport.

The contract is **optional-by-default** and parsed **tolerantly** (unknown keys ignored, missing fields defaulted, discriminators stored as pass-through strings). That is what lets a partially-supported league render a thinner-but-correct screen instead of crashing (Part IV §5).

## 5. Capability flags — the render-or-hide gate

The three discriminators pick the *renderer*; they do **not** encode which optional datum a given league actually has. Roughly 40+ elements across the screens are per-sport-gated (win prob, situation, power play, form, weather, seeds, summary tier…). Rather than re-litigate "hide win-prob for soccer/hockey/golf/tennis/cricket/racing" on three screens, resolve a small **capability-flag table through the same `extends` chain as the discriminators**, sourced from the guide's presence data:

| Flag | True for | Gates |
|---|---|---|
| `hasSummaryTier` | all except golf/mma/tennis/racing | Box / Plays / Stats chips; the whole rich detail |
| `hasSituation` | baseball, basketball (cheap) | bases/count/outs, bonus, cheap live win-prob |
| `hasWinProb` | baseball, basketball, football (summary/core) | the win-probability bar |
| `hasScoringPlaysArray` | football only | the `scoringPlays[]` feed shape (others derive from `plays[]`/`keyEvents[]`) |
| `hasPlaysFeed` | australian-football, baseball, basketball, hockey | dense play-by-play tab |
| `hasCommentary` | soccer, cricket (+ curated) | soccer narrative / cricket ball-by-ball |
| `hasForm` | soccer, rugby, rugby-league | the `WLWWL` form string (others derive from schedule) |
| `hasPowerPlay` | hockey | PP badge/clock |
| `hasLineScores` | most team sports (not soccer/field-hockey) | per-period grid |
| `hasSeeds` | basketball (NCAA, core), tennis (`curatedRank`) | bracket seed badges |
| `hasWeather` | baseball, australian-football (~1%) | weather chip |
| `rankingsFeed` | `polls` (college) / `tour` (ATP-WTA) / `divisions` (UFC) / none | the standalone rankings page |

Every "derived / NOT OBSERVED / per-sport-gated" element below becomes one deterministic lookup: **render if the flag is set, else hide the element cleanly** (never an empty chip).

## 6. Shared building blocks (used by every screen — defined once)

These read identically off `scoreboard` competitors and `summary.header` competitors, so define them once and reference them from every screen table.

| Block | Field path (scoreboard) | Availability / notes |
|---|---|---|
| **Team identity** | `…competitors[].team.color` / `.alternateColor` / `.abbreviation` / `.displayName` / `.shortDisplayName` / `.logo` | color 92%, logo 98%, names/abbr 100% — **team sports only**. Athlete sports (golf/mma/racing) carry `competitors[].athlete.displayName` + `athlete.flag.alt/.href` instead of a club logo/color. |
| **Home/away & loser-dim** | `…competitors[].homeAway` (59%), `.winner` | drives two-row order (`USA vs BRA`, `@` for away). Loser row dims by numeric score compare. |
| **Score** | `…competitors[].score` (STRING, 73% once live) | print per `scoreKind`. |
| **Status pill / phase** | `…status.type.name` (100%), `.type.state`, `.type.shortDetail`, `.displayClock`, `.period` | **branch phase on `type.name`, not `state`.** `shortDetail` gives `BOT 7` / `Q3 8:22` / `73'` / `Final` / `9:40 PM`; `.type.altDetail` gives the `OT`/`SO`/`F/10` suffix. |
| **Venue** | `…competitions[].venue.fullName` (100%), `.venue.address.city` (96%) | cheap — no summary/core fetch needed for the name. Racing uses `events[].circuit.fullName`; tennis uses `groupings[].competitions[].venue.court`. |
| **Team record** | `…competitors[].records[].summary` (type `total`, 64%) | fallback → standings core path. |
| **Series pips + context** | `…competitions[].series.competitors[].wins` + `.id`, `series.totalCompetitions`, `series.summary` | series 4% (postseason baseball/basketball/hockey/soccer). `series.title` is generic (`Playoff Series`) — **for the round/`GAME 6` label use `competitions[].notes[].headline`** (100% of note objects, e.g. `East 1st Round - Game 6`). "Game N" = `sum(wins)+1`; "can clinch" = `max(wins)+1 == ceil(total/2)` — both **derived, no field.** |
| **Poll rank badge** | `…competitors[].curatedRank.current` (19%; `99`→unranked) | college AP/coaches; **not** a bracket seed. |
| **Athlete headshot** | inline on scoreboard: `situation.batter/pitcher.athlete.headshot` (100%), `competitors[].probables[].athlete.headshot` (100%), `competitors[].leaders[].leaders[].athlete.headshot` (86%), `status.featuredAthletes[].athlete.headshot` (99%) | **prefer these cheap inline sources**; the standalone core-athlete `headshot.href` is only 33–56%. Build `a.espncdn.com/i/headshots/{league}/players/full/{id}.png` from the id as a last resort. |

**Assets note (branch on `competitorKind`):** `team` → logo + color bar; `athlete` → headshot + country flag (no logo/color exists). Warm a `team.id → {logo, logoDark, color}` cache from scoreboard/`/teams` on first sight, because **standings and summary payloads lack team color** and standings needs the join. Provide an a11y fallback (tricode-on-neutral chip) when color is missing (8%) or two teams share a color.

# PART II — PER-SCREEN DATA CONTRACTS

Each screen lists only the data that is *specific* to it; the shared blocks (identity, status, score, series) from Part I §6 are assumed. `[verified]` marks a path the adversarial pass confirmed; `⚠` marks a correction it forced; `NOT OBSERVED` marks a datum with no ESPN source.

## 2.1 Home Feed (Scores tab)

**Endpoints:** one `scoreboard` per followed league + per favorite (cheap, polled). Date-strip "has games" dots from a wide `?dates=YYYYMMDD-YYYYMMDD` range scoreboard scan (or `core-calendar/ondays` per league). **The whole feed must stay on the cheap tier.**

**One renderer, three discriminators + presence.** `competitorKind` picks the identity cell (team row vs athlete event block); `layout` picks the card body (two rows vs leaderboard); `scoreKind` prints the score; the glance flourish dispatches on which optional object is present (`situation`, `details[]`, `series`, `probables`, `curatedRank`).

| UI element | Tier | Field path | Availability / fallback |
|---|---|---|---|
| Stacked favorite hero + league game rows (identity, score, status, venue) | scoreboard | *Part I §6 shared blocks* | — |
| Baseball bases diamond | scoreboard | `situation.onFirst / .onSecond / .onThird` | ~86% of **live baseball**; `situation` is baseball+basketball only. Empty diamond if absent. |
| Baseball count / outs / batter | scoreboard | `situation.balls / .strikes / .outs`, `situation.batter.athlete.displayName / .summary` | live baseball ~86% / batter 64%. |
| Probable pitchers/goalies (`Gilbert vs Webb`) | scoreboard | `competitors[].probables[].athlete.displayName / .shortName / .headshot`, `.record` (`(5-4, 3.88)`), `.statistics[]` (W/L/ERA) | 11% — baseball (SP) + hockey (goalie, `status.type=='confirmed'`); pre-game only. |
| Soccer live minute (`73'`) | scoreboard | `status.displayClock` (98%, `90'+6'`), `status.type.shortDetail` | — |
| Soccer red-card note (`NED down to 10`) | scoreboard | `details[].redCard` + `.team.id` (tally reds per team → 11−reds) `[verified derive]` | `details[]` present on live/final soccer. |
| Soccer penalty result (`1 (4) / 1 (2)`, `MEX advance`) | scoreboard | `competitors[].shootoutScore`, `status.type.name==STATUS_FINAL_PEN`, `competitors[].advance`, `notes[].text` | shootoutScore ~1%; `notes[].text` = `X advance 4-3 on penalties`. |
| Soccer cheap goal/card mini-timeline | scoreboard | `details[].type.text / .clock.displayValue / .scoringPlay / .athletesInvolved[].displayName` | the cheap equivalent of the summary timeline; **empty in a 0-0**. |
| Playoff series pips / context / `GAME 6` / `can clinch` | scoreboard | *Part I §6 series block* (`notes[].headline` for the label; wins/total for pips; clinch derived) | ⚠ not `series.title` (generic). |
| NBA possession arrow / `in bonus` / timeouts | core | `situation` (`homeFouls.bonusState`, `homeTimeouts.timeoutsRemainingCurrent`) or infer possession from `situation.lastPlay.team.id` | ⚠ **NOT on scoreboard** → hide on the feed (fetch only on detail open). |
| NBA/other **win probability** (`68%`) | core / scoreboard(NBA) | NBA cheap: `situation.lastPlay.probability.homeWinPercentage` (14%). Others: core `predictor`/`probabilities` | ⚠ **no win-prob on the cheap tier except basketball** → omit on the feed for baseball/football/hockey/soccer, or accept a per-card core fetch. |
| Football down&distance / yardline / drive bar | core / summary | core `situation` (`down`, `distance`, `yardLine`, `isRedZone`); drive geometry from summary `drives.previous[].plays[].start.*` | ⚠ **NOT on scoreboard** (situation is baseball+basketball only) and **no `drives.current`** in the guide — the live drive bar is core-only. Show a static down/distance chip from core `situation`; forgo the animated bar on the feed. |
| Round/competition label (`WORLD CUP QF`) | scoreboard | soccer `competitions[].altGameNote` (24%, `FIFA World Cup, Round of 16`); else registry league name + `weekLabel` | — |
| `Winner faces FRA/ESP` bracket-forward line | core | `core-tournaments…/bracketology` `winnerAdvancesTo` | NOT on scoreboard (`notes[].text` holds results only) → omit or use a cached bracket. |
| Header `TODAY` + full date; date-strip weekday/day labels | device | client-formats the selected date; `day.date` (98%) confirms the fetched slate | no `TODAY` string exists. |
| Date-strip per-day "has games" dot | calendar | `core-calendar/ondays.eventDate.dates[]` per league, or a range `?dates=` scan; `leagues[].calendar[]` on scoreboard (96%) is a **hint only** (sparse for MLB) | cross-check actual events before dimming a day. |
| League section header + `See all N` | device/scoreboard | registry name/route (primary); `leagues[].name`/`.abbreviation` echo; `N` = `events[].length` | — |

**Gaps:** live NFL/CFB glance (down&distance, drive, possession) can't be glanceable from the poll — core-only. Win-prob on the feed is basketball-only. Hockey PP badge, NBA bonus, forward bracket context are not on the poll. College-football scoreboard needs `?groups={id}&limit=400` to return a full slate (a **registry data quirk**, not a code branch).

## 2.9 Venue & Circuit (stadium facts · F1 circuit map)

**Design refs:** `Venue.dc.html` (14a — Wrigley Field) and `Circuit.dc.html` (13a — Spa-Francorchamps). A game-detail **tab**, not a top-level screen: the same event header + `ChipNav` as every other detail, then a photo/map card, then the shared 2-col `StatCell` fact grid. It is the same card grammar for a stadium and an F1 circuit — which is exactly why it's **one tab that dispatches on data presence**, not two sport-named screens.

**Endpoints:** the header (score/status/city), `indoor`, and `attendance` are on the **cheap scoreboard already in hand** (`competitions[].venue`, `competitions[].attendance`, `events[].circuit`). # PART III — CROSS-CUTTING & PLATFORM

These concerns are owned by no single screen but every screen depends on them. Building them once (rather than per-screen) is what keeps the app coherent.

## 3.1 Assets & identity module

A single primitive that every screen calls. **Warm a `team.id → {logo, logoDark, color, altColor, abbreviation}` cache** from the scoreboard poll and `/teams`, because standings and summary payloads **omit team color** and standings needs the join. Logos have a dark-background variant (`/teams` `logos[].rel` includes `dark`; scoreboard `team.logo` → derive the `/500-dark/` URL) — use it in the dark UI and fall back to `logo` on 404. **Branch on `competitorKind`:** `team` → logo+color bar; `athlete` → `headshot` + `athlete.flag` (country), because golf/tennis/mma/racing carry **neither** a club logo nor a color. Athlete headshots are best sourced inline from the cheap scoreboard (Part I §6), not the 33–56% core-athlete doc. a11y: label/logo fallback when color is null (8%) or two teams share a color.

## 3.2 League catalog & per-league data quirks (the registry)

The app knows which leagues exist and how to route them from the **on-device registry** (`league-profiles.json`, ~245 leagues), not a fetch. Promote per-league **data quirks** into that registry so they never become sport-name code branches:

- **College football** scoreboard needs `?groups={id}&limit=400` or it under-returns the slate.
- **Calendar mode** per league: `day`-type (NBA/NHL/EPL → per-day dots) vs `list`-type (NFL/CFB → week windows, no per-day dots).
- **Tennis anomalous path**: `events[].groupings[].competitions[]` and `competition.id != event.id`.
- Each league's verified `espnLeagueId` (string) + `verifiedIds` honesty flag.

## 3.3 Calendar / date coverage

The date strip's "has games" dots come from either a wide `?dates=YYYYMMDD-YYYYMMDD` range scoreboard scan (cheapest, works everywhere) or per-league `core-calendar/ondays.eventDate.dates[]`. The scoreboard's own `leagues[].calendar[]` (96%) is a **hint, not truth** — density is non-uniform (dense per-game-day for NBA/NHL/EPL; sparse season-boundary markers for MLB), so **cross-check actual `events[]` before dimming a day empty.**

## 3.4 Odds / weather / injuries / news / broadcast — out-of-band & media scope

One consolidated decision so three screens don't re-discover the same holes:

| Data | Reality | Decision |
|---|---|---|
| **Broadcast / TV badge** | `competitions[].broadcast` (100%) + `geoBroadcasts[].media.shortName` (ESPN+/Paramount+/…) on the **cheap scoreboard** | **add to game rows/hero** — one line, near-free (currently unmapped). |
| **Odds** (spread/O-U) | inline on ~25% of scoreboard events, near game time, baseball/basketball/football/soccer only | render when present; game-detail pre-game block = core `competition-odds`; per-team **moneyline is core-only** or parse `odds[].details` string. |

| **Injuries** | site `/injuries` returned empty; **core `teams/{id}/injuries` DID serve** (deref `items[].$ref` → status/type/returnDate, AF/baseball/basketball/hockey) | best-effort via **core**; report as available-via-core (empty for some teams), not absent. |

# PART IV — DISCOVERY METHODOLOGY (sports/leagues the design never drew)

The design draws ~12 leagues; the registry carries ~245 across 17 sports, and ESPN adds more. Treat "a league the app was never designed for" as the **normal case** and light it up **without new per-sport UI code**, via five mechanisms in order.

## IV.1 Classify into the discriminators (the rendering contract)

Classify any league into `layout · scoreKind · competitorKind` (+ `periodUnit`/`regulationPeriods`) from the **cheap scoreboard alone** — exactly what `probe.mjs` fingerprints:

| Discriminator | Observable signal | Decision |
|---|---|---|
| `layout` | count of `competitors[]` per competition | 2 → `headToHead`; N>2 ordered by `order` → `field` |
| `layout` trap | `competitors[].homeAway` present | team H2H — but tennis/MMA are still `headToHead` with `homeAway` present-but-meaningless (decide on the 2-count) |
| `scoreKind` | `competitor.score` samples | ints → `numeric`; `-10`/`E`/`+3` → `toPar`; `161/5 (…)` → `cricket`; no scalar → `none` |
| `competitorKind` | identity shape | `team{}` → `team`; single `athlete`, no team → `athlete`; `type='team'` + 2-entry roster → `pair` |

Classify it and it **inherits every existing renderer** — the score card, the line-score grid, the situation dispatch, the standings table — with zero new UI. Path traps: tennis under `groupings[]`; golf/MMA per-unit scores under nested `competitors[].linescores[].linescores[]`; never assume `competitions.length===1`.

## IV.2 Register it as DATA (three-tier `extends` chain)

`league-profiles.json` is `families` → `profiles` → `leagues`, resolved by one walker (`resolve.mjs`; Dart-ported on-device, verified byte-for-byte). Nearest value wins; scalars replace, objects shallow-merge. Adding a league is **one JSON object**:

```json
"soccer/xyz.1": { "extends": "soccer.domestic", "espnLeagueId": "<verified>", "name": "…", "abbr": "…", "region": "…", "priority": "v3", "verifiedIds": true }
```

**Where to attach:** (1) find the family whose discriminators + period model already match → `extends` it with just the verified id + naming; (2) if several leagues share the same deltas, factor an intermediate profile (`golf.strokeplay`); (3) mint a **new family** only when the `layout·scoreKind·competitorKind`/`periodUnit` combination doesn't yet exist. The registry is bundled as an app asset; `buildCatalog` runs on-device, so a new league flows automatically into the catalog, Explore, the normalizer's profile resolution, and the feed — no fetch, no code touched.

## IV.3 Discover the data surface from the evidence guide

`schema/espn-guide/` is ground truth (crawled, not guessed). Read in order:

1. **Support matrix** (`by-sport/index.md`, `index.md`) — does the tier even exist? (summary 404s for golf/mma/tennis/racing; rankings only basketball/football/mma/tennis; cricket has no teams directory.)
2. **Per-sport guide** (`by-sport/<sport>.md`) — the leagues + their shape, which endpoints work / 404 (with URL templates), reachable core resources, and the sport's distinctive **fingerprint** fields.
3. **Per-endpoint field tables** (`scoreboard.md`, `summary.md`, `standings.md`, `core-*.md`) — read `presence%` (render only when present), `sports` (a field only lights up for the listed sports), and `type` (`str-numeric`).

**Evidence discipline (load-bearing):** every cited field path must appear in the guide; if a datum isn't there, write `NOT OBSERVED — <source needed>` rather than inventing a path. Regenerate the guide (`crawl.mjs` → `rollup.mjs`) when it goes stale.

## IV.4 Onboard a genuinely novel sport

Run the `onboard-league` workflow: **Probe** (`probe.mjs --deep` → fingerprint) → **Propose** (classify against families, draft the minimal `extends` entry, `mode` ∈ existing-drift / new-league / new-family / novel-no-data) → **Verify** (an adversarial agent re-fetches live and tries to break the id/period/scoreKind/edge-cases) → **Finalize** (paste-ready `leagues{}` entry + any `canonical.ts` change + `SCHEMA.md` note + `residualGaps`; `ready:false` if anything's unverified). Keep it honest afterward with `verify.mjs` on CI/cron (CRITICAL on id drift / dead slug → exit 1; WARN on shape drift / unmapped status name) and `--snapshot`/`--diff-snapshot` to catch ESPN silently restructuring.

## IV.5 Degrade gracefully (render-time)

The discriminator-driven UI renders whatever a league has and omits the rest:

- **Tolerant parsing** — optional-by-default contract; unknown keys ignored, missing fields defaulted; never hard-require a tier/field.
- **Unknown status names** → `phase:'unknown'` pass-through (raw `detail` still shows); `verify.mjs` flags it to be mapped later. Branch on `status.type.name`, not `state`.
- **Missing summary** (golf/mma/tennis/racing) → build detail from scoreboard + core; the line-score grid renders from cheap `periodScores` first, upgrades to `summary.periodLines` if it arrives.
- **Situation card dispatches on presence** → a novel sport with none of the signals simply shows no card (correct, not a special case).
- **Absent standings/groups** → offseason returns `groups:[]` (renders nothing); ungrouped → one `''` group; team not in a table → omit `standing`.
- **Calendar is a hint** — cross-check events before dimming a day.

**Invariants:** no code branches on sport name; adding a league is one JSON object; every cited path is observed; the three tiers are checked before use; standings is always core `apis/v2`; scores are strings; the core graph is discovered via `$ref`; honesty is automated; everything degrades, nothing crashes.
